## 项目约定（维护与升级）

### 总体原则

- 先保证“最小可用闭环”，再做增强；每次改动尽量可回滚、可验证。
- 前端与后端的核心业务术语保持一致（例如：单词、词库、反馈、到期复习）。
- 任何持久化数据结构变更都必须通过数据库迁移（Prisma migration），禁止手改线上库。

### 目录与命名

- 前端页面放在 `src/pages/`，复用组件放在 `src/components/`，纯逻辑放在 `src/lib/`，数据放在 `src/data/`。
- 后端路由统一挂在 `/api/*`，控制器命名为 `xxx.controller.ts`。
- 业务变量/函数命名允许中文（本项目以中文为主），但对外接口字段尽量保持稳定、语义明确。

### 版本与兼容

- API 路由与字段尽量向后兼容；必须破坏兼容时，新增版本前缀（例如 `/api/v2/...`）。
- 前端本地存储 key 需要带版本号（例如 `xxx.v1`），升级时写迁移逻辑或保底降级策略。

### 数据一致性（无登录）

- 通过请求头 `x-device-id` 区分设备；后端若未收到该头，会在响应头回传生成的 `x-device-id`。
- 前端需要持久化保存 `x-device-id`（localStorage 即可），并在每次请求时附带。
